# v0.31.0 Release Notes (in-flight) — Frontier tier added; planning + verification re-tiered onto Fable 5

Covers the org change "Fable does assessment + planning + verification; Opus coordinates; Sonnet executes." Adds the `frontier` tier to `model_overrides.py` (default `fable`), retunes role pins across the verification surface, keeps a documented Sonnet exception for high-frequency advisory checkers.

## Why

Wrong plans and wrong verification verdicts compound — a bad plan dispatches N implementers into the wrong work; a missed verdict ships a regression. User's standing priority is Accuracy > Speed > Cost (`feedback_accuracy_speed_cost_priority.md`). The planning + verification surfaces — where one miscall poisons everything downstream — now pay the Frontier premium. Execution and coordination stay where they were because they are either bounded application (Sonnet implementer applies a settled plan) or routing (Opus orchestrator chooses which subagent runs next, with deterministic gates as the safety net).

## Deliberate Sonnet exceptions on the verification surface (documented)

Two verification-shaped agents stay on Sonnet rather than escalating to Fable. The tension with round-2 evidence ("rubric-application = Sonnet is robust") is real; the user chose Fable for the rest of the verification surface because the compound-risk profile dominates per-call cost, but high-frequency advisory checkers stay cheap.

- `alignment-checker` (Sonnet) — called per queue item during autonomous iterate (up to 25× per run). Advisory only, doesn't gate. Cost dominates value at this fan-out frequency.
- `synthesis-critic` (Sonnet) — per-UI-commit WARN-only check. Advisory only, never gates. Frequency × non-gating shape means a cheaper tier is the right tradeoff.

Pins are defaults, not locks — these can be lifted if telemetry says so.

## Execution escalation target is Opus, not Fable

Fable is reserved for planning and verification verdicts. Execution under genuine ambiguity is a coordination call (interpret the spec, route to a new chunk, decide whether to re-plan) and stays on Opus. If the surfaced ambiguity is actually a **planning miss** (the plan no longer fits), route back to Phase 2 — Fable re-plans, then execution resumes on Sonnet/Opus.

## Planning-tier correction note (Advisor trio, `experiment/advisor-judge-trio`)

> This release row says "Fable runs Phase 1 Assess + Phase 2 Plan synthesis." A 2026-06-10 assessment found that **dormant** at release: no Fable-pinned planner agent existed, both `build-orchestrator` and `assessment-orchestrator` are `model: opus`, and a subagent can't change its own model mid-run — so plan synthesis actually ran **inline on the orchestrator's model (Opus in agent dispatch; the session model in `/build-loop:run`)**. Fable touched plans only as a WARN-only reviewer *after* the plan existed.
>
> The **Advisor trio** experiment makes the claim true *conditionally*: Phase 2 plan synthesis runs at Frontier (Fable) **when stakes-gating trips** (`synthesisDensity > 5`, `riskSurfaceChange`, `stakes >= medium`, or `dispatch_tier: frontier`) via the 4-rung **Advisor dispatch ladder** — `advisor` agent (Rung 1), peer host (Rung 2), or an already-Fable session (Rung 0). When no trigger fires or no dispatch path is reachable, the orchestrator synthesizes inline and **labels it honestly** (Rung 3 = today's behavior; the floor equals current state, no regression). Read "Fable plans Phase 1/2" as *the aspiration the ladder enforces on high-stakes plans*, not an unconditional guarantee. Full protocol: `skills/build-loop/references/advisor-dispatch-ladder.md`.
